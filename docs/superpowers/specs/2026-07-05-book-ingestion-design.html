Book Ingestion, Grounded to the Theorem

date 2026-07-05 status approved, pending final review depends on c636533 restructure · 432b426 HAS_DOCUMENT fix
Approved

§1Goal

Ingest books into the same Neo4j graph as papers, with grounding fit for a mathematician: answers cite “Theorem 3.1, Cont & Tankov, §3.2, p. 71”, and a concept extracted from a book resolves to the same global Concept node as the identical concept extracted from a paper.

Books were explicitly deferred by the original document-graph design because sequential per-chunk extraction times out on book-length inputs, and the paper triage stage quarantines anything that is not a paper. This design removes both obstacles without touching the working paper path.

§2Decisions

QuestionDecision
Grounding granularityTheorem / definition level. Numbered results and definitions become first-class nodes carrying book, chapter, section, and page location.
Extraction coverageEverything, chapter-partitioned. The whole book is eventually extracted; one Dagster partition per chapter makes it restartable and observable.
CorpusBorn-digital publisher PDFs. Clean text layers, usually embedded TOC bookmarks. OCR out of scope.
RoutingSeparate BOOKS_SOURCE_DIR watched by its own sensor. No paper-vs-book auto-triage.
ArchitectureApproach A — parallel asset lineage, shared internals. A second asset chain in the same code location, reusing existing packages as libraries.

Alternatives rejected

  • Branch inside existing assets (one partition set, doc_type flag): couples both pipelines, keeps the book-timeout problem, and puts regression risk on a paper path that works today.
  • Separate Dagster code location or repo sharing only Neo4j: forces packaging or duplicating the resolution stack — and duplicated entity resolution is exactly how two drifting Concept spaces happen.

§3Graph schema

New node labels: Chapter and Section. Everything else is reused: Book (declared in schema.py, never yet populated), Document, Chunk, Concept, Definition, Result, Author.

// nodes
(:Book    {id, title, authors, year, edition, isbn, publisher})
(:Chapter {id, number, title, page_start, page_end})   // id = {book_id}:ch03
(:Section {id, number, title, page_start, page_end})   // number = "3.2"

// edges
(Book)-[:HAS_DOCUMENT]->(Document)       // same anchor pattern as Paper (432b426)
(Author)-[:AUTHORED]->(Book)
(Book)-[:HAS_CHAPTER {order}]->(Chapter)
(Chapter)-[:HAS_SECTION {order}]->(Section)
(Chunk)-[:BELONGS_TO]->(Document)        // unchanged — existing chunk RAG intact
(Chunk)-[:PART_OF]->(Section)            // new — every chunk located in hierarchy
(Section)-[:STATES]->(Definition|Result) // papers: Paper-STATES
(Definition)-[:DEFINES]->(Concept)       // identical to papers
(Result)-[:USES]->(Concept)              // identical to papers
(Result)-[:DEPENDS_ON]->(Result)         // may cross chapters within a book
(Book)-[:COVERS]->(Concept)              // declared long ago, finally populated

Definition and Result gain a label property carrying the book’s own numbering ("Theorem 3.1"; null when unnumbered) and a page property. Ids stay content-local, {book_id}:ch03:res:{hash12}, mirroring the paper-local scheme so statements never falsely merge across sources.

Concepts are the bridge between corpora. “Every statement of the Lévy–Khintchine formula I have” is one pattern, returning Papers and Sections mixed:

MATCH (c:Concept {name: "Lévy-Khintchine representation"})
      <-[:DEFINES|USES]-(s)<-[:STATES]-(loc)
RETURN s.label, s.statement, loc   // loc: mix of Paper and Section nodes

Housekeeping folded in

  • Add Chapter, Section, Document to NODE_TYPES; add HAS_DOCUMENT, HAS_CHAPTER, HAS_SECTION, PART_OF to RELATIONSHIP_TYPES (the paper pipeline’s HAS_DOCUMENT is missing from the list today).
  • Unique constraints on Chapter.id, Section.id; extend PATTERNS with the new triples.
  • Fix stale pipeline.schema / pipeline.cypher imports in scripts/init_neo4j.py and scripts/reset_graph.py, broken since the restructure.

§4Identity

book_id = isbn:{isbn13} when frontmatter extraction finds one, else title:{normalized} — the same precedence idiom as compute_paper_id (doi > arxiv > title). No Semantic Scholar, no external lookup in v1. Chapter and section ids derive from book_id plus position.

§5Asset chain & partitioning

Two dynamic partition sets, two jobs, one new sensor. New assets live in pipeline/assets/; logic in a new pipeline/books/ package plus the reused existing packages.

Lane 1 · books partitions — key = SHA-256 of the PDF · sensor on BOOKS_SOURCE_DIR

→
book_parsedpypdfium2, per-page text + PDF outline
→
book_metadataLLM frontmatter → title, ISBN, book_id

book_structureChapter/Section tree; registers ch. partitions ⤵
→
book_chunkssplit_markdown per section + embeddings
→
book_structure_writeBook/Chapter/Section/Chunk → Neo4j
✓ after this single run, vector RAG covers the whole book — before any extraction
Lane 2 · book_chapters partitions — key = {book_sha}:ch03 · auto-enqueued, serialized

→
book_chapter_resolvedthe same resolve_concepts() as papers
→
book_chapter_graph_writeSection-STATES · COVERS · alias_map upserts
✓ same canonical_key · same pgvector thresholds (0.90 / 0.60) · same LLM adjudicator · same audit table

The “a Lévy process is a Lévy process” guarantee is enforced by code reuse, not convention: book chapters call the identical resolve_concepts() against the identical Postgres entity_embeddings and alias_map tables that papers use. There is one Concept space.

Orchestration

  • After book_structure_write succeeds, all chapter partitions are auto-enqueued. The existing max_concurrent_runs=1 serializes them, preserving the single-writer invariant resolution depends on.
  • Chapters run in ascending order: cross-chapter DEPENDS_ON back-references resolve at write time by (book_id, label); a forward reference to a not-yet-extracted chapter is logged and skipped in v1.
  • The paper chain is untouched. Shared-file edits are limited to schema.py additions and small refactors that expose existing functions with no paper-path behavior change.

§6Error handling

  • Scanned / no text layer → QuarantineError("needs-ocr") via the existing needs_ocr() check.
  • No outline and heading detection fails → QuarantineError("no-structure") with a log of what was tried. Corpus is born-digital, so rare; the v2 fallback would be synthetic fixed-size sections, not a redesign.
  • Chapter extraction/write failure → that partition fails visibly in the Dagster UI and re-runs alone. All graph writes are MERGE-based on content-hash ids, so re-runs are idempotent.
  • Forward DEPENDS_ON → logged and skipped.
  • A half-extracted book is a valid state, not an error: structure + RAG for everything, theorem graph for extracted chapters. The Dagster partition view is the progress bar.

§7Testing

  • Unit: outline → Chapter/Section tree builder (messy-nesting fixtures); section-aware chunking with correct page attribution; label parsing (“Theorem 3.1”, “Definition 2.14”, unnumbered); book_id precedence (isbn > title).
  • Integration fixture: a small LaTeX-built book PDF (~3 chapters, a few numbered theorems) checked into the repo; full chain against dev Neo4j; assert hierarchy edges, Section-STATES, labels, pages via Cypher.
  • The Lévy-process test, literally: seed a Concept via the paper path, extract a fixture chapter mentioning the same concept, assert both sources point at the same Concept node — the shared-resolution guarantee as a regression test.
  • Paper-path regression: existing tests untouched and passing.

§8Out of scope (v1)

  • OCR / scanned books
  • The retrieval/query layer (spec-only for papers too; this schema is designed to serve both when built — spec/05-query-router.md)
  • Cross-source statement linking (book theorem ↔ paper theorem)
  • External metadata enrichment (Open Library, etc.)
  • A book analogue of paper_analysis summaries
  • Figures and exercises
