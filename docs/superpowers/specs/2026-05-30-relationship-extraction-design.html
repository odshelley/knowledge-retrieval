Relationship Extraction — Design

Date: 2026-05-30  ·  Status: Approved (brainstorming) → ready for implementation plan

Scope: Add inter-entity relationships to the bespoke document-graph builder so the Neo4j graph carries the connective tissue its schema declares.

1 · Problem

The pipeline extracts typed entities per paper — Concept, Definition, Result — and writes them to Neo4j. But today those entities are islands: Definition and Result nodes hang off their Paper via STATES and nothing else; Concept nodes are shared across papers but carry no relationships among themselves or to the definitions/results that use them.

pipeline/schema.py already declares the missing edges in PATTERNS (Definition–DEFINES→Concept, Result–USES→Concept, Result–DEPENDS_ON→Result, Concept–RELATED_TO→Concept), and the docstring claims extraction validates against them — but no asset writes them and extraction never produces the links in the first place.

Consequence: the multi-hop GraphRAG queries the graph exists for — "which theorems use concept X", "what does Theorem 3.1 depend on", "which concepts relate to BSDE" — cannot be answered.

Why bespoke

The pipeline moved off neo4j-graphrag's SimpleKGPipeline for better entity-identity discipline (content-hash ids, conservative cross-paper concept resolution, idempotent MERGEs, citation backfill). That trade inadvertently dropped relationship extraction. This work restores it on the stronger entity foundation.

2 · Locked decisions

  • Target schema = academic / alethograph (schema.py): Paper, Author, Concept, Definition, Result, Summary, Chunk, Document (+ deferred Book, Topic, Researcher, Idea). The spec/01+spec/03 bank-quant-domain schema is an earlier, abandoned design, not the target (though its prompt prototyped the hybrid pattern adopted here).
  • Topic linking & Idea construction are downstream, out of this pipeline. The pipeline leaves the seam: canonical Concept nodes, reachable concept↔concept structure, accessible embeddings. Topic nodes are never written during ingestion.
  • Relationship model = hybrid: small typed predicate + free-text description + confidence (validated by prior art, §6).
  • Purpose = multi-hop GraphRAG retrieval (+ feeding the downstream Idea skill).

3 · Scope & phasing

Two phases. Both committed; Phase 2 is deferred for risk-sequencing, not optional.

Phase 1 — build now within-paper edges

EdgeEndpointsRisk
Definition –DEFINES→ Conceptpaper-local Definition → canonical Conceptlow
Result –USES→ Conceptpaper-local Result → canonical Conceptlow
Result –DEPENDS_ON→ Resultboth paper-local Resultslow

Lower risk: each edge has ≥1 paper-local endpoint and needs no cross-paper edge merge.

Phase 2 — committed, next concept ↔ concept hybrid edges

Typed predicate (SPECIALIZES, RELATED_TO, COMPARES_WITH, OTHER) + description + confidence + provenance (support_count, source_paper_ids). Higher risk: both endpoints canonical → recurs across papers → idempotent cross-paper merge + keep-both-on-conflict. Full sketch in §5. The extraction-schema change is designed once to cover both phases.

4 · Phase 1 design

4.1 Core idea

The LLM sees only text, never our content-hash ids. So extraction emits links by name; graph_write — the single authority on identity — translates names to canonical concepts / result-ids and writes the edges. Unresolvable references are dropped, never invented (faithfulness gate).

extracted_graph (LLM)            resolved_entities          graph_write
  definitions[i].defines ──┐      surface→canonical   ──┐    build maps:
  results[j].uses          │  →   (concept names)       │ →   surface→canonical
  results[j].depends_on  ──┘                            │     result-name→result_id
                                                         │    3 new MERGE passes
                                                         │    unmatched name → skip+count

4.2 Extraction schema + prompt  extraction.py + extraction_anthropic.py

  • definitions[].defines : list[str] — concept names this definition defines (usually one).
  • results[].uses : list[str] — concept names this result invokes.
  • results[].depends_on : list[str] — result names this result depends on (e.g. ["Lemma 2.4"]).

Prompt gains: "For each definition, list which extracted concepts it defines. For each result, list which extracted concepts it uses and which other results (by name) it depends on. Reference ONLY entities you already listed; if unsure, omit." — the faithfulness gate.

4.3 Dataclasses + dedup  extraction.py

  • Definition gains defines: list[str]; Result gains uses and depends_on (default []).
  • parse_extraction reads new fields (default []).
  • merge_results unions link lists when deduping an entity seen in overlapping chunks. Dedup keys unchanged.

4.4 resolved_entities change

Emit the surface name alongside the canonical so graph_write can map references:

resolved.append({
    "surface": c["name"],   # NEW: original extracted name, for link resolution
    "name": canonical, "kind": c["kind"], "action": action.value, "embedding": v,
})

4.5 graph_write — maps + three MERGE passes

surface_to_canon  = {c["surface"]: c["name"] for c in concepts}
name_to_result_id = {r["name"]: r["id"] for r in rrows if r["name"]}
-- DEFINES: paper-local Definition → canonical Concept
MATCH (d:Definition {id:$def_id}), (c:Concept {name:$canon}) MERGE (d)-[:DEFINES]->(c)
-- USES: paper-local Result → canonical Concept
MATCH (r:Result {id:$res_id}), (c:Concept {name:$canon}) MERGE (r)-[:USES]->(c)
-- DEPENDS_ON: Result → Result (both paper-local)
MATCH (r1:Result {id:$res_id}), (r2:Result {id:$dep_id}) MERGE (r1)-[:DEPENDS_ON]->(r2)

Names that don't resolve (a concept/result the LLM didn't actually list) are skipped and counted under a skipped_refs metric — surfaced, never fabricated.

4.6 Idempotency

Free: endpoints have stable ids (content-hash for Def/Result, canonical name for Concept) and every edge is a MERGE. Every Phase 1 endpoint is paper-local or already-canonical → no cross-paper edge merge needed.

4.7 Schema changes

None. DEFINES/USES/DEPENDS_ON are already in RELATIONSHIP_TYPES and all three patterns already in PATTERNS. Phase 1 simply populates edges the schema promised.

4.8 Tests

  • Unit: parse_extraction reads/defaults new fields; merge_results unions link lists; graph_write map-building + unmatched-ref skip/count + edge rows.
  • Integration: extend test_one_paper_end_to_end to assert the new edges + endpoints; extend idempotency test to assert edge counts stable on re-run.

5 · Phase 2 design sketch committed roadmap

This phase is planned and will be built. Recorded here so picking it up is "execute the plan", not "redesign". Risk-sequenced after Phase 1.

5.1 Extraction

# concept_relations[]: {src, dst, predicate, description, confidence}
#   predicate ∈ {SPECIALIZES, RELATED_TO, COMPARES_WITH, OTHER}

EDC-style: extract freely, canonicalize to the vocabulary; OTHER retains the raw predicate. USES is excluded from the concept-concept set (stays the Result→Concept edge). Keep all extractions above a low confidence floor — confidence is a ranking signal, not a hard gate.

5.2 Schema additions (Phase 2 only)

  • RELATIONSHIP_TYPES: add SPECIALIZES, COMPARES_WITH.
  • PATTERNS: add (Concept, SPECIALIZES, Concept), (Concept, COMPARES_WITH, Concept) (RELATED_TO already present).

5.3 Edge model

(c1:Concept)-[:SPECIALIZES {
    description: "...",        // the specific sense
    confidence: 0.9,          // true per-extraction confidence — NOT frequency
    support_count: 3,         // number of papers asserting this edge
    source_paper_ids: [...]   // provenance, dedup-appended
}]->(c2:Concept)

5.4 Cross-paper merge mechanics

  • Edge identity = (src_canonical, predicate, dst_canonical); volatile fields in ON CREATE/ON MATCH, never the MERGE pattern → idempotent under Dagster retries.
  • Both endpoints resolve to canonical concepts → connects concepts across papers.
  • Keep-both on conflict: a different predicate between the same pair is a distinct edge; attribute via source_paper_ids.
  • Symmetric RELATED_TO: pre-sort the endpoint pair so (A,B)/(B,A) collapse; query undirected.
  • Aggregate on match: dedupe-append source_paper_ids, increment support_count only for a new paper, keep confidence as max — distinct from count.
  • Provenance: arrays now; reify as :Evidence/MENTIONED_IN only if they grow large (deferred).

6 · Research grounding

Four parallel agents surveyed prior art (2026-05-30). Hybrid is established best practice, not novel:

  • EDC — Extract, Define, Canonicalize (EMNLP 2024): the canonical hybrid — extract freely, snap to a typed predicate when it fits, retain free-text otherwise. arxiv.org/abs/2404.03868
  • SciERC (Luan et al., 2018): proven small scientific vocabulary — USED-FOR, HYPONYM-OF (≈ SPECIALIZES), PART-OF, COMPARE (≈ COMPARES_WITH), FEATURE-OF, EVALUATE-FOR; hierarchy kept distinct from association. aclanthology.org/D18-1360
  • Neo4j SimpleKGPipeline: typed predicates via patterns with relationship properties (description, confidence); resolve-then-MERGE-on-canonical-id keeps endpoints canonical. docs
  • MS GraphRAG / LightRAG: merge mechanics — key on endpoints(+predicate), volatile fields out of MERGE, accumulate provenance, summarize descriptions when they pile up, symmetric via sorted pairs. GraphRAG · LightRAG

Refinements over a naive port of spec/03

  1. Keep confidence separate from frequency — store support_count distinctly from a true per-extraction confidence.
  2. Keep-both-with-attribution on cross-paper conflicts rather than overwriting.
  3. Faithfulness gate: reference only listed entities / allow abstention to suppress LLM over-prediction (Hallucination-Resistant RE, 2025).

7 · Optional follow-ups (not Phase 1 scope)

  • Enforce PATTERNS at write time: graph_write hard-codes edges and never consults PATTERNS; a one-line guard asserting each written edge is in PATTERNS would make the contract real.
  • Tighten schema.py docstring to state the vocabulary is consumed by tests + bootstrap and (until the guard exists) is not a runtime-enforced extraction gate.
